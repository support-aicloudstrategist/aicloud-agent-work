Case 1 — D2C e-commerce, AWS, ~₹35L/month spend

Context: 120-person company on AWS ap-south-1 for three years. CFO flagged a 38% year-on-year increase in cloud spend that did not track revenue growth (which grew 11% in the same window). No prior FinOps review.

Findings: 41% of EC2 compute on idle instances after a Diwali-season scale-up that was never reversed (m5.4xlarge fleet of 22, averaging 8% CPU). 47TB in S3 Standard that had not been queried in 14 months (Glacier candidate). 38 orphaned EBS volumes totalling 4.2TB. Reserved Instance coverage at 12% on a steady-state workload. No resource tagging, so cost-per-team was unknowable.

Action (4 weeks, one senior engineer): Right-sized 60% of EC2 fleet to m5.xlarge / m5.2xlarge based on p95 CPU + memory. S3 lifecycle policies moved cold data to Glacier Instant Retrieval. Cleanup automation for orphaned EBS + snapshot retention policy (30 days). RI + Savings Plan blend sized for 80% baseline coverage with 3-year No-Upfront commitment. Infracost wired into GitHub PR pipeline. OpenCost deployed for EKS workload visibility. Tagging enforced via SCP.

Outcome: Verified ₹9.4L/month savings (27% of bill) by end of week 5, confirmed against the following two billing cycles. Ongoing cost baseline now owned by engineering — CFO receives a monthly cost-per-order metric instead of a line-item bill.

Case 2 — B2B SaaS, multi-region GCP, ~₹18L/month spend

Context: 45-person Series A company running a multi-tenant analytics product on GKE across asia-south1 and us-central1. The US region existed only because the first three customers were US-based; 80% of load had since shifted to India. Cost-per-tenant was not measured.

Findings: GKE node pools provisioned for peak QPS that occurred twice a quarter (customer-triggered reports), leaving 71% headroom 24×7. Cross-region egress of ~3.4TB/month because the analytics warehouse sat in us-central1 but queries originated in asia-south1. 18 persistent disks unattached for 90+ days (2.1TB). No committed-use discounts despite three years of stable baseline. Cloud SQL instance oversized by 4× after a historical load test that was never rolled back.

Action (6 weeks): Cluster autoscaler tuned with PodDisruptionBudgets + custom HPA on actual QPS. Replicated analytics warehouse to asia-south1 with scheduled sync (egress dropped 94%). Disk reclamation cron with 7-day quarantine before deletion. 1-year committed-use discount on the India baseline. Cloud SQL right-sized after 30 days of observation. Cost-per-tenant dashboard built on BigQuery billing export.

Outcome: 27% cloud-bill reduction (~₹4.9L/month) sustained for 8+ months with zero customer-facing incidents during the transition. Engineering OKR now includes cost-per-tenant ceiling; new feature PRs include a cost impact estimate from Infracost.

Case 3 — AI-first startup, AWS with GPU fleet, ~₹22L/month spend

Context: 30-person seed-to-Series-A generative AI product built on AWS. Inference workload on SageMaker real-time endpoints, fine-tuning jobs on p4d.24xlarge On-Demand, feature store backed by DynamoDB. GPU spend grew 4× in two quarters as the product scaled, with no cost-per-inference visibility. Leadership had no answer for the board when the AI infrastructure line item crossed ₹20L/month.

Findings: Two real-time inference endpoints provisioned on g5.2xlarge instances serving 3 req/s average, 12 req/s peak — latency SLA met by g5.xlarge at half the cost. Fine-tuning jobs running on On-Demand p4d instances when spot capacity in us-east-1 had 85% availability at 70% discount. No inference caching for a summarisation endpoint where 40% of requests had identical prompts in a 24-hour window. DynamoDB feature store in provisioned capacity mode running at 11% utilisation — on-demand billing would have been cheaper at that pattern. Model artifacts in S3 Standard when 95% of files were inference-only reads of the latest version.

Action (4 weeks): Right-sized real-time endpoints to g5.xlarge with autoscaling (min 1, max 4). Migrated 80% of fine-tuning workloads to spot p4d with checkpointing (recovery tested). Added a Redis semantic cache for the summarisation endpoint. Switched DynamoDB to on-demand. S3 lifecycle rules moved older model artifacts to Glacier Instant Retrieval. Built cost-per-1K-inferences dashboard per endpoint, tagged by product feature.

Outcome: 41% AI infrastructure reduction (~₹9.1L/month) verified across 3 billing cycles. GPU idle time dropped from 47% to under 10%. Cost-per-1K-inferences now tracked against product feature margins; the build-vs-buy decision for a new RAG feature was made with actual numbers for the first time.

Case 4 — Fintech platform, AWS, DevOps & platform engineering

Context: 55-person Series B fintech serving Indian SMBs with a payments and reconciliation product. Deployments took 45–90 minutes, required a senior engineer on-call, and were informally banned on Fridays. New engineers took 5–6 weeks to reach their first production deploy. No staging environment existed in any meaningful form — "staging" had drifted from production for over two years. MTTR on production incidents averaged 2.4 hours.

Findings: CI pipeline running a monolithic 34-minute test suite with zero caching or parallelism. Deployments executed via a shell script on a bastion host — no IaC for the deployment mechanism itself. Terraform coverage at 52% of production resources; the rest was a mix of CloudFormation remnants, console edits, and one engineer's personal Ansible playbook. Secrets in environment variables baked into AMIs. Kubernetes adopted 18 months earlier for one service, never adopted for the rest — cluster running underutilised. No policy-as-code: seven production deployments in the last quarter had shipped with `:latest` image tags.

Action (4 weeks): GitHub Actions pipelines rebuilt with matrix parallelism and aggressive caching (CI time dropped from 34 to 8 minutes). ArgoCD deployed, five services migrated to GitOps with progressive rollout. Terraform module library built — coverage raised from 52% to 94%. AWS Secrets Manager integration; secrets pulled from environment variables. OPA Gatekeeper policies on the cluster. Infracost integrated into PRs. Internal developer CLI (devctl) for service scaffolding and preview environments. Two enablement sessions with the full engineering team, including a live rollback rehearsal.

Outcome (measured 60 days post-delivery): Lead time for changes reduced from 9 days to 4 hours. Deployment frequency increased from 2 per week to 11 per week. Change failure rate dropped from 24% to 8%. MTTR fell from 2.4 hours to 38 minutes. Team moved from DORA "Low" tier to "High" tier on three of the four metrics. Friday deploys became routine.
